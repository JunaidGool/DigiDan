Somewhere in the marketing for every message broker is the phrase "exactly-once delivery." It's the feature everyone wants and almost nobody actually gets. Distributed systems run on unreliable networks, and an unreliable network forces a hard choice: deliver a message _at most_ once and risk losing it, or _at least_ once and risk sending it twice. For anything that matters — a payment, a debit order, a ledger entry — losing it is unacceptable, so you choose at-least-once. Which means: sooner or later, the same message arrives twice.

The redeliveries aren't exotic. A consumer processes a message and then crashes before it can acknowledge; the broker, having heard no ack, redelivers. A network blip makes the ack time out; redelivered. A consumer group rebalances mid-batch; redelivered. A producer retries a publish it wasn't sure had landed; now there are two. None of these are bugs — they're the system working as designed. In a payments flow, each one is a real person's money being moved twice.

So we don't chase exactly-once _delivery_. We engineer exactly-once _effect_, and we move the guarantee off the transport and into the consumer, where we actually control it.

## Idempotency is the whole game

An operation is idempotent if doing it twice produces the same result as doing it once. `Set the balance to R500` is idempotent — run it a hundred times and the balance is R500. `Subtract R100` is not — run it twice and you've taken R200. The moment a consumer's job is naturally idempotent, duplicate delivery stops being a correctness problem and becomes a performance footnote.

Most real financial operations aren't naturally idempotent, so we make them idempotent. That comes down to a few disciplines.

## A stable key that comes from the business, not the broker

A consumer first needs a reliable way to recognise a message it has already handled. The broker's own message id is the wrong tool — it can change on redelivery or republish, so the "same" event turns up wearing a different id. Instead the key comes from the _event_: a payment id, a transaction reference, or a deterministic hash of the fields that define the operation. The producer stamps this **idempotency key** onto the event, and the consumer keys every dedup decision off it. Same business event, same key, every time — no matter how often or how many ways it's delivered.

## The dedup check and the effect in one transaction

This is the part that's easy to get subtly wrong. The naïve version reads "have I seen this key?", and if not, does the work. Two redeliveries race each other, both read "no," and both do the work. The check and the effect have to be **atomic**, or they aren't a guarantee at all.

The clean way is to let the database enforce it. Put a unique constraint on the idempotency key and insert it in the _same_ transaction as the ledger write:

```sql
begin;
  insert into processed_events (idempotency_key) values ($key)
  on conflict (idempotency_key) do nothing;   -- duplicate? inserts nothing

  -- only a real effect when the insert above won the race
  update ledger set balance = balance - $amount where account = $acct;
commit;
```

If the insert hits the unique constraint, this is a duplicate: the transaction becomes a no-op — or, better, returns the result you stored the first time. The database, which already gives you ACID guarantees for the ledger, becomes the thing that also gives you exactly-once. You're not inventing a distributed protocol; you're leaning on the strongest consistency boundary you already have.

Where an operation spans more than one row or has to be serialised per entity — everything touching one account, say — a **distributed lock**, or a single-writer partition (keying the stream by account so one consumer owns it), keeps concurrent related events from interleaving.

## Store the outcome, not just the fact

For anything that answers a caller, record the _result_ of the first processing next to the key, not merely that it happened. Then a duplicate returns the original outcome instead of a bare "already done." That's the difference between a system that's idempotent internally and one that's idempotent all the way out to whoever asked.

## Duplicates are not the only failure

Idempotency solves redelivery. It does not solve **ordering** — a duplicate and an out-of-order message are different problems. If event B can overtake event A, you need sequence numbers, versioning, or operations designed to commute. Don't let idempotency lull you into ignoring it.

And some messages fail every time — malformed, or referencing something that no longer exists. A **poison message** must not retry forever and wedge the stream. After a bounded number of attempts it goes to a dead-letter queue, raises an alert, and the pipeline keeps moving. Retrying is not a strategy for a message that will never succeed.

## The ledger is the backstop

Everything above is the online path. The reason we can trust it in a financial system is that it isn't the only line of defence. The effects land in an **append-only ledger with an audit trail**, and an end-of-day **reconciliation** re-derives the truth and checks it against what the system recorded. Deterministic replay means state can be rebuilt from the event log and compared. If a duplicate ever did slip through, reconciliation is designed to catch it: the online idempotency keeps the books right in real time, and the reconciliation proves they're right.

## The honest posture

"Exactly-once" is not a checkbox on a broker. It's a property you build in the consumer, by assuming the opposite — that every message will arrive at least twice, at the worst possible moment, in the wrong order. Idempotency is what makes that assumption boring. In a system that moves money, boring is the entire goal.
